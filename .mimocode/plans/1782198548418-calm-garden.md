# Fix: Appointments Not Appearing Across All Deployments

## Root Cause

The `department-appointments` endpoint (`server/routes/appointment.js:1140`) has a **branching bug** that causes inconsistent behavior:

### How the endpoint works today:

```
if (user's department is in GENERAL_DOCTOR_DEPARTMENT_KEYS):
    → Return ALL active appointments ✅
else:
    → Return ONLY the logged-in user's own assigned appointments ❌
```

### Why some deployments show all appointments, others don't:

The `GENERAL_DOCTOR_DEPARTMENT_KEYS` set (line 22) contains:
```
'general', 'generaldentistry', 'oral', 'oralmedicine', 'oralmedicineandradiology', 'oralmedicineradiology'
```

The frontend route's `allowedDepartments` (in App.jsx) includes additional departments:
```
'oralandmaxillofacial', 'oralandmaxillofacialsurgery'
```

**These are NOT in the backend set.** So any user whose department is `oralandmaxillofacial` or `oralandmaxillofacialsurgery`:
1. ✅ Can access the page (frontend allows it)
2. ❌ Falls into the `else` branch (backend doesn't recognize it as "general")
3. ❌ Only sees their own assigned appointments (not all department appointments)

Additionally, even within "general" departments, the endpoint **only** returns appointments with status in `["pending", "assigned", "confirmed", "in_progress", "reschedule_requested", "rescheduled"]` — excluding `completed` and `closed` appointments.

### The 4 deployments testing scenario:

When you test with 4 different users across 4 deployments:
- Users whose department matches the `GENERAL_DOCTOR_DEPARTMENT_KEYS` → see ALL appointments
- Users whose department doesn't match → only see their own appointments

This explains the inconsistent behavior you're seeing.

---

## Fix

### File: `server/routes/appointment.js` (lines 1140-1211)

**Remove the branching logic entirely.** The `department-appointments` endpoint is specifically designed for the Oral Medicine Department ecosystem view — it should return ALL active appointments for ANY authorized user, regardless of their department.

The frontend `ProtectedRoute` already handles department-based access control. The backend should trust that access decision and return all data.

### Change:

Replace the entire endpoint body (lines 1140-1211) with simplified logic:

```javascript
router.get("/department-appointments", auth, requireRole(["doctor", "chief-doctor", "pg", "ug"]), async (req, res) => {
  try {
    const appointments = await Appointment.find({
      status: { $in: ["pending", "assigned", "confirmed", "in_progress", "reschedule_requested", "rescheduled"] },
    }).sort({ appointmentDate: -1, appointmentTime: -1 });

    const enriched = await attachPatientName(appointments);
    res.json({ success: true, appointments: enriched });
  } catch (err) {
    console.error("Error fetching department appointments:", err);
    res.status(500).json({ success: false, message: 'Failed to fetch department appointments' });
  }
});
```

This removes:
- The `isGeneralDept` check (line 1144)
- The entire `else` branch (lines 1152-1202) with its complex doctor-key matching logic
- The `normalizeDepartment` dependency for this endpoint

---

## Why This Is Safe

1. **Access control is already handled** by:
   - Frontend: `ProtectedRoute` with `allowedRoles` + `allowedDepartments`
   - Backend: `requireRole(["doctor", "chief-doctor", "pg", "ug"])`
   
2. **No sensitive data exposure**: Appointments contain patient names, dates, and complaints — all of which should be visible to authorized department staff.

3. **The endpoint name literally means "department appointments"** — it should return all of them.
