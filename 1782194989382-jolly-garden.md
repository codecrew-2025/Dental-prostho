# Plan: Cross-Deployment Appointment Sync & Oral Medicine Department Dashboard

## Problem Analysis

**Root Cause**: All 4 deployments share the same MongoDB Atlas cluster and database — so the data IS shared. The problem is that existing appointment endpoints are **role-scoped**:

- `GET /my-appointments` → only returns appointments assigned to the logged-in doctor
- `GET /pg-appointments` → only returns appointments assigned to the logged-in PG
- `GET /all-appointments` → returns ALL appointments, but requires `admin`/`chief`/`chief-doctor` role

When a patient books an appointment on Deployment A and it gets assigned to Doctor X, a different doctor logging in on Deployment B won't see it under "my-appointments" because it's not assigned to them. There is **no centralized department-level view** that shows all appointments for the Oral Medicine department.

**Secondary Issue**: The existing `OralMedicine.jsx` is a case-sheet form, not an appointment dashboard.

## Solution Overview

Create a new **Oral Medicine Department Appointment Dashboard** that fetches ALL appointments from MongoDB and displays them in a shared, filterable view — accessible to all authorized staff in the department.

### No MongoDB changes needed — the data is already shared across deployments.

Create a new page `OralMedicineDashboard.jsx` that fetches ALL appointments from MongoDB and displays them in a shared, filterable view. This page will be accessible to doctors/PGs/chief-doctors in the oral medicine department.

**Backend**: Two options:
- **Option A (simpler)**: Relax the role check on `GET /api/appointment/all-appointments` to also allow `doctor` and `pg` roles — but this would expose ALL appointments across ALL departments, which may not be desired.
- **Option B (recommended)**: Add a new endpoint `GET /api/appointment/department-appointments` that returns all appointments where the assigned doctor belongs to the requesting user's department. This keeps data scoped per department while still showing all appointments within that department.

**Frontend**: New page with:
- Real-time-style refresh (poll every 30 seconds or manual refresh button)
- Filter by: date, status, doctor, patient name
- Show all appointments regardless of which device/deployment created them
- Role-gated access (doctor, pg, chief-doctor in oral medicine department)

## Implementation Details

### OralMedicineDashboard.jsx

- On mount: fetch `GET /api/appointment/all-appointments` (for chief-doctor) or `GET /api/appointment/my-appointments` (for regular doctors)
- Display as a table with columns: Booking ID, Patient Name, Patient ID, Date, Time, Chief Complaint, Status, Assigned Doctor
- Auto-refresh every 30 seconds (configurable) to catch new appointments from other deployments
- Manual "Refresh" button
- Filters: date range, status dropdown, search by patient name/ID
- Department guard: only accessible to users with department `oralmedicine` (or similar variants)

### Route Configuration (App.jsx)

```jsx
<ProtectedRoute allowedRoles={["doctor", "pg", "ug", "chief-doctor"]} 
                allowedDepartments={["oralmedicine", "oralmedicineandradiology"]}>
  <OralMedicineDashboard />
</ProtectedRoute>
```

### Auto-Refresh Pattern

```javascript
useEffect(() => {
  fetchAppointments();
  const interval = setInterval(fetchAppointments, 30000); // 30s
  return () => clearInterval(interval);
}, []);
```

## Notes

- The `all-appointments` endpoint already exists and returns all appointments from the shared MongoDB — no new backend endpoint is strictly needed
- The 30-second polling is a pragmatic solution; for true real-time you'd need WebSockets, but that's a larger architectural change
- If you want different departments to each have their own dashboard, the same pattern can be replicated for Pedodontics, Conservative Dentistry, etc.
